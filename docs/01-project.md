# 01 — Проєкт: цілі, рівні, рішення

## Навіщо це існує

Власник — ФОП 3 групи (IT, єдиний податок 5%) — хоче **самостійно вести податки без бухгалтера**:
розуміти, чи є борги/штрафи, чи все подано вчасно, і згодом сам подавати й платити.
Замість того щоб щоразу вручну заходити в кабінет, ми будуємо **власну програму**, яка робить це
через офіційний **API Електронного кабінету ДПС**, підписуючи запити КЕП власника локально.

## Три рівні (roadmap)

### Рівень 1 — ПЕРЕВІРКА (готово ✅)
Прочитати з ДПС усе про власника: борг, стан розрахунків з бюджетом, перелік поданих декларацій,
вхідні/вихідні документи. Реалізовано у `app/clients/dps/` (`Dps::Client`). Лише GET-запити — безпечно.

### Рівень 2 — ЗРУЧНІСТЬ/АВТОМАТИЗАЦІЯ (частково ⏳)
- Гарний звіт замість сирого JSON (таблиця/дашборд).
- Нагадування про дедлайни (напр. «N днів до декларації за півріччя»).
- Порахувати суму до сплати з поквартального доходу.

### Рівень 3 — ПОДАЧА + ОПЛАТА (не зроблено ❌)
- Сформувати XML податкової декларації ФОП 3 групи (форма **F0103309**, наростаючим підсумком).
- Підписати її КЕП і надіслати `POST` на `/cabinet/public/api/exchange` (`/report`), отримати квитанції.
- Оплата: ДПС грошей не приймає — платіж іде **через API банку** (monobank/ПриватБанк)
  на бюджетний IBAN, який беремо з `/ta/splatp`.
- ⚠️ Це незворотні дії — обов'язкове підтвердження власника перед кожною подачею/оплатою.

## Ключові архітектурні рішення та їх причини

| Рішення | Чому саме так |
|---|---|
| **Файловий КЕП `.jks` (ПриватБанк), а не хмарний** | Хмарний підпис (monobank/SmartID) вимагає підтвердження в застосунку на КОЖЕН підпис — його не можна делегувати програмі. Файловий ключ підписує локально без участі людини. |
| **Бібліотека `jkurwa` + `gost89`** | Відкрита JS-реалізація ДСТУ 4145/7564, читає `.jks`/`Key-6.dat`, робить CAdES-підпис. Node вже стоїть на машині. Альтернатива (ІІТ EUSignCP) — пропрієтарна й важча. |
| **Ключ лежить локально, підпис локальний** | Безпека: приватний ключ = юридичний підпис власника; він не передається нікуди, зокрема агенту. |
| **Авторизація = підписаний РНОКПП у заголовку** | Так влаштований приватний API ДПС (див. `03-dps-api.md`). Це не OAuth-пароль, а CMS-підпис. |
| **Клієнти — чистий Ruby, підпис — Node через shell** | ДСТУ 4145 не має практичної Ruby-реалізації, тож `Dps::KepSigner` кличе `lib/kep/sign.js` (jkurwa). HTTP і розбір — на Ruby, тестуються WebMock. |
| **Підпис ДПС БЕЗ `signingTime`** | З атрибутом часу ДПС відхиляє підпис («хибний підпис»). Емпірично встановлено. |

## Історія роботи (як ми сюди прийшли)

1. Почали з питання «як зрозуміти, чи є борги» → пояснили кабінет ДПС і ручний шлях.
2. Власник захотів робити все **через API власною програмою**.
3. Дослідили: приватний API ДПС існує й документований; авторизація — підписаний РНОКПП;
   є готові приклади (`telecomprofi/cabinet-tax-gov-ua-API-curl`) і бібліотека `jkurwa`.
4. Власник згенерував **файловий КЕП у ПриватБанку** (`pb_3525511937.jks` + `sign.crt`) і поклав у папку.
5. Налаштували `jkurwa`+`gost89`, перевірили, що ключ відкривається паролем.
6. Зібрали `Authorization` (підпис РНОКПП), зробили перший запит `/ta/debt` → **HTTP 200**, проаналізували статус.
7. Підключили **monobank Personal API**, звірили дохід із деклараціями.
8. Створили **Rails 8.1 застосунок** (цей проєкт) з повним стеком.
9. **Перенесли** Node-логіку в Ruby-клієнти (`app/clients/`), покрили RSpec, стару папку `app/` видалили.
10. Виявили й полагодили баг підпису: ДПС відхиляв підпис з атрибутом `signingTime` → прибрали `time`.
