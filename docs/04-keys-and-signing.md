# 04 — Ключі КЕП та підпис ДСТУ 4145

## Типи ключів (і чому в нас саме файловий)

| Тип | Носій | Придатний для програми? |
|---|---|---|
| **Хмарний** (monobank / ПриватБанк SmartID / Дія.Підпис) | на серверах провайдера | ❌ ні — кожен підпис вимагає підтвердження в застосунку |
| **Файловий** (`Key-6.dat`, `pb.jks`, `.pfx`) | файл на диску + пароль | ✅ так — підписує локально, автономно |

Тому для автоматизації власник згенерував **файловий КЕП у ПриватБанку**:
- `../pb_3525511937.jks` — контейнер закритого ключа (JKS, захищений паролем);
- `../sign.crt` — сертифікат (публічна частина);
- пароль — у `../password.txt`.

(усі секрети лежать у батьківській теці `~/Documents/taxmate/`, поза Rails-проєктом)

Файловий ключ ПриватБанку видають у Privat24: *Всі послуги → Бізнес → «Завантажити сертифікат»*,
обрати збереження **у файл** (НЕ SmartID). Через monobank файловий ключ отримати НЕ можна (тільки хмарний).

## Сертифікат власника (з `sign.crt`, публічне)

- Власник: `КУЛУЄВ АНДРІЙ УРАЛОВИЧ`, `serialNumber=TINUA-3525511937`
- Видавець: `КНЕДП АЦСК АТ КБ "ПРИВАТБАНК"`
- Чинний: **04.07.2026 – 05.07.2027** (оновлюється раз на 1–2 роки, безкоштовно в Privat24)

> ⚠️ Коли сертифікат протермінується — програма перестане авторизуватись. Треба перевипустити
> ключ у Privat24, замінити `../pb_*.jks`/`../sign.crt` і оновити `../password.txt`.

## Технологічний стек підпису

- **Node.js** (вже стоїть, v22).
- **`jkurwa`** — читає `.jks`/`Key-6.dat`, робить підпис/шифрування за ДСТУ 4145/7564.
- **`gost89`** — криптопримітиви (хеш/шифр ГОСТ/ДСТУ), потрібна залежність `jkurwa`.
- **`jksreader`** — парсер JKS (тягнеться як залежність).

Встановлення (вже зроблено в `app/`):
```bash
npm install jkurwa gost89
```

## Як завантажити ключ і підписати (робочий патерн)

```js
const fs   = require('fs');
const jk   = require('jkurwa');
const algo = require('gost89/lib/compat').algos();   // ОБОВ'ЯЗКОВО передати algo у Box

const box = new jk.Box({ algo });
box.load({
  keyBuffers:  [fs.readFileSync('../pb_3525511937.jks')],
  certBuffers: [fs.readFileSync('../sign.crt')],
  password:    fs.readFileSync('../password.txt', 'utf8').replace(/\r?\n$/, ''),
});

// перевірка, що пара ключ+сертифікат для підпису знайдена:
box.keyFor('sign');                                  // кине помилку, якщо пароль/ключ не той

// внутрішній підпис РНОКПП → DER → base64 (це значення Authorization):
// 🔴 БЕЗ time: — інакше ДПС віддає "хибний підпис".
const der  = await box.pipe(Buffer.from('3525511937'), [{ op: 'sign' }], {});
const auth = Buffer.from(der).toString('base64');
```

### Підводні камені (з нашого досвіду)

- Без `algo` у `new Box({ algo })` буде помилка `algos.hash is not a function`.
- У `.jks` ПриватБанку зазвичай **дві пари** (ключ підпису + ключ шифрування/печатки) — `keyFor('sign')`
  сам обирає підписний.
- Для `Authorization` потрібен **внутрішній** підпис (дані всередині, `detached:false` — це дефолт).
  Прапорець `tax:true` у `pipe` — то вже транспортний формат для **подачі документів**, не для авторизації.
- 🔴 **КРИТИЧНО: підписувати БЕЗ атрибута `signingTime`** — тобто `pipe(data, [{op:'sign'}])`
  **без** `time:`. Якщо додати `time: new Date()`, ДПС відхиляє підпис з `"Помилка перевірки
  підпису: хибний підпис"` (HTTP 400). Саме так налаштовано в `lib/kep/sign.js`.
- Реалізація підпису — у `lib/kep/sign.js`; Ruby-обгортка — `Dps::KepSigner`
  (`app/clients/dps/kep_signer.rb`), клич через shell.

## Для Рівня 3 (подача декларацій) знадобиться

- Сформувати **XML** декларації за схемою ДПС (форма F0103309, коректні коди/періоди).
- Підписати транспортним форматом (`pipe(xml, [{op:'sign', tax:true, …}], …)`), можливо з TSP/OCSP.
- `POST` на `…/exchange/report` з `contentBase64`, потім забрати квитанції (`/kvt_by_id`).
- Спершу відпрацювати на **тестовому сервері** `:9443`, не на бойовому.
